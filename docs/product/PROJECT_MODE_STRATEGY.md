## 1. 배경과 목적

July Canvas는 지금까지 **"아이디어 한 줄 → 브리프/시장조사/제품화전략 초안"** 흐름(아이디어 제품화의 앞단)만 구현되어 있다. 하지만 실제 기획 업무는 두 가지 출발점에서 시작한다.

1. **아이디어에서 출발** — 머릿속 서비스 구상을 시장 검증과 제품화 전략으로 구체화하고 싶다.
2. **요구사항/RFP에서 출발** — 이미 발주처/내부에서 정의된 요구사항·제안요청서가 있고, 이를 기획 산출물과 개발 전달물로 정리하고 싶다.

이 두 출발점은 **입력의 성격, 초기 산출물, AI가 해야 할 일**이 다르다. 본 문서는 July Canvas가 지원할 **두 가지 프로젝트 모드**를 정의하고, 두 모드가 **공유하는 핵심 정책**(특히 IA/기능정의서 역작성)과 **최종 산출물 패키지** 규격을 확정한다.

> 본 문서는 정책을 확정할 뿐 구현하지 않는다. 실제 코드(모드 선택 UI, 분기 로직, 역작성 파이프라인)는 후속 단계에서 단계별 커밋으로 진행한다.

## 2. 두 가지 프로젝트 모드

| | **모드 A — 아이디어 제품화** | **모드 B — 요구사항/RFP 기반** |
|---|---|---|
| 출발점 | 자유 서술형 아이디어 1칸 | 요구사항 정의서 / RFP / 제안 문서 |
| 핵심 질문 | "이게 시장에서 통할까? 어떻게 제품화할까?" | "이 요구사항을 어떤 서비스로 설계하고 전달할까?" |
| 초기 AI 역할 | 시장·제품 관점 **발산**(조사·전략 초안 생성) | 요구사항 **수렴·정리**(분석·기획초안 도출) |
| 초기 산출물 | 브리프 / 시장조사 / 제품화전략 | 요구사항 분석 / 서비스 기획 초안 / 레퍼런스 |
| 공통 중간 단계 | **프로토타입 제작 → 피드백 → 확정** | (동일) |
| 공통 후반 단계 | **확정 프로토타입 기반 IA/기능정의서 역작성** | (동일) |
| 최종 산출물 | **개발 전달용 MD 패키지(3~4종)** | (동일) |

두 모드는 **앞단(입력·초기 문서)만 다르고, 프로토타입 이후 단계는 동일**하다. 즉 "프로토타입 확정"이 두 모드가 합류하는 지점이며, 그 이후는 단일 파이프라인이다.

## 3. 모드 A — 아이디어 제품화 플로우

```
1) 아이디어 입력            자유 서술형 1칸 (현재 ProjectActivationWizard A1 구조)
2) 시장조사                AI 초안: market_research (외부 지식 + 추정 표기)
3) 제품화 전략             AI 초안: product_strategy
4) MVP 방향                제품화전략 내 MVP 범위 / 우선순위 확정
5) 프로토타입              ScreenEditor에서 화면 제작 (HTML) + 정책 주석
6) 피드백 → 확정           댓글/주석 반영 → 프로토타입 확정(lock)
─────────────── 여기서부터 모드 공통 ───────────────
7) IA / 기능정의서 역작성   확정 프로토타입 기반 (§5)
8) PRD + 개발 전달 패키지   최종 MD 3~4종 (§6)
```

초기(2~4단계) 생성물은 **brief / market_research / product_strategy** 3종으로, 현재 `buildActivationDocuments`가 이미 만드는 문서들이다. 모드 A는 현재 구현의 자연스러운 연장이다.

## 4. 모드 B — 요구사항/RFP 기반 플로우

```
1) 요구사항/RFP 입력        요구사항 정의서 / RFP 텍스트·문서 (다중 레퍼런스 허용)
2) 요구사항 분석           AI 초안: 요구사항을 구조화·분류·우선순위화
3) 서비스 기획 초안         AI 초안: 분석을 서비스 기획으로 변환
4) 레퍼런스 정리           경쟁/유사 서비스, 참고 UI, 제약사항 정리
5) 프로토타입              ScreenEditor에서 화면 제작 + 정책 주석
6) 피드백 → 확정           댓글/주석 반영 → 프로토타입 확정(lock)
─────────────── 여기서부터 모드 공통 ───────────────
7) IA / 기능정의서 역작성   확정 프로토타입 기반 (§5)
8) 개발 전달용 MD 패키지     최종 MD 3~4종 (§6)
```

모드 B의 초기 3종(요구사항 분석 / 서비스 기획 초안 / 레퍼런스)은 **모드 A의 brief/market_research/product_strategy 슬롯을 재사용**한다(§7 매핑 참조). 즉 새 `DocumentType`을 추가하지 않고, 동일한 3개 슬롯에 **모드별로 다른 의미·템플릿**을 부여한다.

| 슬롯(DocumentType) | 모드 A 의미 | 모드 B 의미 |
|---|---|---|
| `brief` | 프로젝트 브리프 | 요구사항 분석 |
| `market_research` | 시장조사 | 서비스 기획 초안 |
| `product_strategy` | 제품화전략 | 레퍼런스 정리 |

> 표시 타이틀(`DOCUMENT_META`)은 모드에 따라 라벨이 달라질 수 있으나, **저장 스키마(`type` 값)는 동일**하게 유지한다. 이렇게 하면 ProjectDocuments 에디터·PRD 조립·export·권한 게이팅을 분기 없이 재사용할 수 있다.

## 5. 핵심 정책 — IA/기능정의서는 "역작성"한다

**두 모드 공통의 가장 중요한 원칙.**

- **IA와 기능정의서는 아이디어/요구사항만으로 최종 생성하지 않는다.**
- IA/기능정의서는 **확정된 프로토타입의 코드·화면·플로우·정책 주석을 입력**으로 하여 **역작성(reverse-authoring)** 한다.
- 초기 단계(활성화 직후)에는 **brief/market/strategy(모드 A) 또는 요구사항분석/기획초안/레퍼런스(모드 B)까지만** 생성한다. IA/기능정의서는 이 시점에 **빈 상태로 둔다**(현재 동작과 일치).

### 왜 역작성인가
- IA(정보구조)·기능정의서는 "무엇을 만들지"가 화면으로 확정되어야 정확하다. 아이디어 단계에서 만들면 프로토타입과 어긋나 **두 번 일하게** 된다.
- 프로토타입이 곧 "확정된 진실(source of truth)"이고, 개발 전달물은 그 진실을 문서화한 것이어야 한다.

### 역작성 입력 소스 (확정 프로토타입에서 추출)
1. **화면 코드/구조** — `screens.code`(HTML), 화면 목록·순서
2. **정책 주석** — `comments`(annotation source) + `annotation.history`(정책 버전)
3. **플로우** — 화면 간 이동/연결(주석·기획 의도에서 도출)
4. **초기 문서** — brief/strategy(또는 요구사항분석/기획초안)의 의도·범위
5. **활성화 데이터** — `ProjectActivation`(권한 정책, MVP 범위 등)

### 역작성 결과
- `ia`(IA.md): 화면 트리·정보구조·내비게이션·사용자 플로우
- `feature_spec`(FEATURE_SPEC.md): 화면별 기능·정책·권한·예외 처리
- 이 둘이 채워지면 기존 `generatePRD()`가 PRD에 자동 반영(이미 `## 9 IA` / `## 11 기능정의` 슬롯 존재).

> 역작성 파이프라인은 §3 AI 초안 레이어(`/api/generate/*`)를 재사용한다. 프로토타입 확정(lock) 후에만 활성화되도록 게이팅한다. **구현은 후속 단계.**

## 6. 최종 산출물 — 개발 전달용 MD 패키지

최종 단계의 산출물은 KAKE 예시처럼 **개발팀이 그대로 받아 착수할 수 있는 MD 3~4종**으로 전달한다.

| # | 파일 | 내용 | 출처 |
|---|---|---|---|
| 1 | `DEVELOPMENT_HANDOFF.md` | 개발 착수용 종합 핸드오프(개요/구조/우선순위/먼저 볼 파일/검증 기준) | PRD + 제품화전략 + 프로토타입 |
| 2 | `PRD.md` | 제품 요구사항 정의(현행 `generatePRD` 확장) | 활성화 + IA + 기능정의서 + 프로토타입 URL |
| 3 | `USER_APP_UI_SPEC.md` | 사용자 앱 화면/기능/정책 명세 | 역작성 IA/기능정의서의 사용자 영역 |
| 4 | `ADMIN_UI_SPEC.md` | 관리자(어드민) 화면/기능/정책 명세 | 역작성 IA/기능정의서의 관리자 영역 |

규칙:
- **3~4종 가변** — 관리자 영역이 없는 프로젝트는 `ADMIN_UI_SPEC.md`를 생략해 3종으로 전달.
- 모든 산출물은 **확정 프로토타입 기반**이어야 한다(§5). 프로토타입 미확정 상태에서는 핸드오프 패키지를 생성하지 않는다.
- 산출물은 기존 **export(MD/PDF/PPTX)** 와 예약된 **`outputs` 컬렉션(`OutputType`)** 을 토대로 패키징한다. `prd_md`/`ia_md`/`feature_spec_md`/`prototype_url`이 이미 예약돼 있어 `DEVELOPMENT_HANDOFF`/`USER_APP_UI_SPEC`/`ADMIN_UI_SPEC`만 추가 검토하면 된다(스키마 확정은 후속 단계).

## 7. 데이터 모델 매핑 (스키마 변경 없이)

본 정책은 **기존 타입을 거의 그대로 재사용**하도록 설계한다.

### 7.1 모드 식별 (구현됨 — M2)
- `ProjectActivation`에 **`mode?: ProjectMode`** optional 필드를 추가해 활성화 데이터 안에 저장한다(별도 컬렉션/스키마 변경 없음). 값:
  - `idea_productization` — 모드 A(아이디어 제품화)
  - `requirement_planning` — 모드 B(요구사항/RFP 기반)
  - `legacy` — mode 미저장 기존 프로젝트(모드 A와 동일 처리)
- 미지정(레거시·KAKE)은 `lib/documents.ts`의 `resolveProjectMode()`가 **모드 A로 폴백**한다.
- 모드는 **활성화 위저드 첫 단계에서 1회 선택**하며 이후 문서 제목·content 프레이밍·입력 문구 분기에만 영향. **DocumentType(6종)은 동일하게 유지**한다.

### 7.2 문서 타입 (변경 없음)
- `DocumentType` 6종 그대로. 모드 B는 brief/market_research/product_strategy 슬롯을 §4 매핑대로 재사용.
- `DOCUMENT_META`의 **표시 타이틀만 모드에 따라 라벨링**(저장 `type` 불변).

### 7.3 상태 머신 (기존 값 활용)
`ProjectStatus = draft | active | review | approved | archived | handoff` — 이미 `review`·`handoff`가 예약돼 있다.

| 상태 | 의미(모드 공통) |
|---|---|
| `draft` | 생성 직후, 모드/아이디어 미입력 |
| `active` | 아이디어/요구사항 입력 + 초기 3종 초안 생성됨 |
| `review` | 프로토타입 제작·피드백 진행 중 |
| `approved` | 프로토타입 확정(lock) + IA/기능정의서 역작성 + PRD 승인 |
| `handoff` | 개발 전달 MD 패키지 생성·전달 완료 |
| `archived` | 보관 |

→ 새 상태값 추가 없이 기존 머신으로 두 모드를 모두 표현. 전환 **조건만** 모드별로 정의(후속 구현).

### 7.4 산출물 (예약 모델 활용)
- `OutputType`에 (후속) `dev_handoff_md` / `user_ui_spec_md` / `admin_ui_spec_md` 추가 검토. `prd_md`/`ia_md`/`feature_spec_md`/`prototype_url`은 이미 존재.

## 9. 구현 영향 / 향후 작업 (이번엔 코드 변경 없음)

본 문서 확정 후, 후속 단계에서 다음을 **각각 독립 커밋 + 검증**으로 진행한다. (FLOW 로드맵 B/C와 정합)

1. **모드 선택 UI** — 활성화 진입 시 모드 A/B 선택. `Project.mode` optional 추가(스키마 영향 최소).
2. **모드 B 초기 문서 템플릿/프롬프트** — 요구사항분석/기획초안/레퍼런스용 스켈레톤 + `/api/generate` 분기.
3. **역작성 파이프라인** — 프로토타입 확정(lock) 후 IA/기능정의서 생성(`/api/generate`, 입력=화면/주석/플로우). 게이팅 필수.
4. **개발 전달 패키지** — DEV_HANDOFF/USER_UI_SPEC/ADMIN_UI_SPEC 생성 + `outputs`/export 패키징 + `handoff` 상태.
5. (FLOW C) QA Automation 연결 — 역작성 기능정의서를 QA 시나리오 입력으로 재사용.

## 10. 변경 금지 / 주의사항 (본 단계)

- **코드·Firestore Rules·Wizard·ProjectDocuments·AI API 변경 금지.** 본 단계는 문서 작성만.
- IA/기능정의서를 **아이디어/요구사항만으로 최종 생성하지 않는다**(역작성 원칙, §5).
- 새 `DocumentType` 추가를 지양하고 **기존 6종 + 슬롯 재사용**으로 두 모드를 표현(§7).
- 모드 도입은 `Project.mode` optional 1필드 수준으로 **스키마 영향 최소화**. 미지정은 모드 A 폴백.
- AI/QA/전달 패키지는 **항상 추가 레이어**, 키 없거나 미확정 시 기존 흐름이 그대로 동작(graceful).
- KAKE 등 레거시 데이터·`.env.local`은 변경/커밋 금지.
