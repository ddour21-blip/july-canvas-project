단순한 명령어 입력창이 아닙니다. AI PM이 사용자의 프롬프트를 확인하고, "과거 GS/계룡 자료를 스캔하겠다"며 방향성을 제시하고 컨펌을 받습니다. 이를 통해 빠른 의사결정이 가능함을 보여줍니다.

샌드박스 실시간 렌더링 (우측 패널)

'집뷰 옵션몰' 예시를 그대로 시뮬레이션했습니다. 상단 메뉴(타입 A/B)가 퍼블리싱되어 있고, 사용자가 바닥재/가전 버튼을 누르면 3D 공간(Mockup)의 상태가 실시간으로 변하는 것을 의사결정자가 직접 테스트해 볼 수 있습니다.

PRD 배포 승인 구조

샌드박스 상단에 [승인 및 PRD 배포] 버튼이 활성화됩니다. 관리자가 눈으로 결과물을 확인하고 승인하면, 즉시 배포 파이프라인이 가동된다는 워크플로우를 완벽하게 증명합니다.

AI Native Company OS 개발 계획



1. 상품 전략 정의



핵심 고객



* 10~100명 규모의 스타트업/콘텐츠/IT 회사

* 레포, 문서, 기획서, 회의록, 디자인 자산이 쌓여 있지만 재활용이 안 되는 조직

* 대표 또는 본부장이 회의 의사결정 병목을 느끼는 회사



핵심 문제



* 과거 프로젝트, 레포, 문서, 담당자를 사람이 기억에 의존해 찾음

* 회의가 감정적 의견, 직급, 목소리 큰 사람 중심으로 흐름

* 회의 후 결정사항, 액션아이템, 근거가 흐려짐

* 기획/개발/디자인/마케팅/제작이 따로 움직임



핵심 가치



* AI가 회의에 참여해 회사 자산을 검색하고 근거 기반 의견을 제시

* 각 파트별 AI가 본부장/시니어 전문가 역할로 토론

* 대표는 최종 승인만 하면 되는 구조

* 회의 결과가 자동으로 실행 가능한 PRD, 개발 계획, 디자인 요청서, 마케팅 계획으로 정리됨



핵심 차별점



* 일반 회의록 AI가 아니라 “회사 기억 + 역할별 AI 의사결정 보조”

* 단순 요약이 아니라 과거 레포/문서/사례를 찾아 의사결정에 연결

* 사람의 감정적 충돌을 중재하고 객관적 근거를 제시

* 개발, 디자인, 콘텐츠, 마케팅까지 실행 계획으로 전환



수익 구조



* 초기: B2B 구축형/파일럿 과금

* 월 구독: 조직당 월 100만~500만 원

* 추가 과금: 연결 소스 수, 회의 시간, AI 역할 수, 저장 문서량 기준

* 엔터프라이즈: 사내 GitHub/Drive/Figma/Slack 연동 구축비 별도



최초 진입 시장



* 내부 회사용 MVP

* 이후 유사한 IT/콘텐츠 제작 회사 대상

* 특히 “과거 자산이 많고, 프로젝트가 자주 바뀌는 조직” 우선

8. 첫 번째 /goal 프롬프트



/goal



이건 실행 요청이 아니라 AI Native Company OS를 만들기 위한 사전 분석이야.



목표:

회의에 AI가 함께 참여해 회사의 과거 자산, 레포, 문서, 기획서, 회의록을 찾고,

각 파트별 AI 본부장이 의견을 제시하며,

감정적 의견을 중재하고,

대표가 최종 의사결정을 할 수 있도록 보고서를 생성하는 MVP를 만들고 싶다.



먼저 현재 프로젝트 구조와 주요 파일을 살펴보고,

내가 /goal로 맡기면 효과가 큰 작업 3개를 추천해줘.



서비스 핵심 방향:



* AI 회의 참여

* 회사 자산 검색

* 파트별 AI 의견 생성

* 의사결정 보조

* 회의 중재

* 액션아이템 정리

* 최종 보고서 생성

* 대표 최종 승인 구조



각 추천 항목에는 아래 내용을 포함해줘.



* 목표

* 왜 지금 하면 좋은지

* 수정 범위

* 완료 기준

* 지켜야 할 제약

* 검증 명령어

* 리스크

* 예상 소요 시간



추가로 아래도 함께 분석해줘.



* 현재 시스템 구조

* 데이터 흐름

* 상태 관리 방식

* 재사용 가능한 기존 코드

* 새로 만들어야 하는 영역

* MVP 기준에서 과한 구조

* 가장 먼저 안정화해야 하는 영역



반드시 아래 규칙을 지켜줘.



* 아직 코드는 수정하지 마

* 아직 파일도 만들지 마

* 실행 전 분석만 해줘

* 기존 동작을 추측하지 말고 실제 파일 기준으로 분석해줘

* 구조를 이해하기 전 리팩토링하지 마

* 수정 범위를 최소화해서 제안해줘



마지막에는 바로 복사해서 사용할 수 있는 /goal 프롬프트 3개를 작성해줘. 

해당 기능 구현을 위해 우선 정책과 방향성을 먼저 정리하고
1차 시뮬레이터 기반으로 제작해서 프로토타입을 공유하여 싱크를 맞춘 후
2차 실제 DB 작업 및 코딩 작업순으로 진행하려고 함
대표가 원하는 의도에 맞춰 해당 기능을 개발할 수 있게 작성해줘

정리 방향은 “지금 바로 개발”이 아니라, 대표/본부장이 같은 그림을 보도록 정책 → 1차 시뮬레이터 → 2차 실제 구현 순서로 고정하는 문서 형태로 잡겠습니다.

19s 동안 생각함
아래는 정책/방향성 정리 → 1차 시뮬레이터 프로토타입 → 2차 실제 DB/코딩 작업 순서로 진행하기 위한 개발 기준 문서 형태입니다.
핵심은 “AI가 회의에 참석해 회사 자산을 찾고, 역할별 의견을 만들고, 대표 승인 후 실행 산출물로 전환되는 구조”를 먼저 눈으로 증명하는 것입니다.

AI Native Company OS 개발 방향 및 1차 시뮬레이터 제작 기준
1. 개발 목적
AI Native Company OS는 단순 회의록 요약 도구가 아니다.

이 제품의 핵심은 회사 내부에 쌓인 레포, 문서, 기획서, 회의록, 디자인 자산, 과거 프로젝트 자료를 AI가 회의 중에 찾아보고, 각 파트별 AI 본부장이 근거 기반 의견을 제시하며, 의사결정권자가 최종 승인하면 실행 가능한 PRD와 작업 계획으로 전환되는 의사결정 OS를 만드는 것이다.

초기 목표는 완전 자동화 시스템이 아니다.
먼저 대표와 본부장이 실제로 이해하고 판단할 수 있는 1차 시뮬레이터 기반 프로토타입을 만든다.

2. 핵심 메시지
이 프로토타입은 다음 메시지를 명확하게 보여줘야 한다.

AI가 단순히 답변하는 것이 아니라 회의 참석자처럼 행동한다.

AI PM이 사용자의 요청을 이해하고, 필요한 과거 자료를 찾겠다고 제안한 뒤 컨펌을 받는다.

회사 자산이 RAG 형태로 검색되고, 기획/디자인/개발/마케팅/QA AI가 역할별로 병렬 판단한다.

중앙 파이프라인을 통해 실제 시스템이 작동하는 것처럼 진행 상태가 보인다.

우측 샌드박스에서 결과물이 실시간으로 시뮬레이션된다.

관리자가 결과물을 확인한 뒤 승인하면 PRD 배포 또는 실행 파이프라인으로 넘어간다.

즉, “회의 → 자산 검색 → AI 본부장 토론 → 샌드박스 검증 → 대표 승인 → 실행 산출물 생성” 흐름을 한 화면에서 증명해야 한다.

3. 1차 개발 원칙
3-1. 1차는 실제 DB 시스템이 아니다
1차 목표는 실제 DB, 실제 RAG, 실제 3D 렌더링, 실제 배포 자동화가 아니다.

1차는 의사결정자에게 제품의 방향성을 설명하기 위한 시뮬레이터다.

실제 옵션몰 3D 대신 UI 상태가 변하는 샌드박스 Mock 사용

3-2. 1차에서 증명해야 하는 것
1차 프로토타입은 기술 완성도가 아니라 제품 개념을 증명해야 한다.

11. 1차 개발 요청용 /goal 프롬프트
/goal

AI Native Company OS의 1차 시뮬레이터 프로토타입을 만들기 위한 사전 분석과 설계를 진행해줘.

이번 작업의 목표는 실제 DB/API/외부 연동을 구현하는 것이 아니라, 대표와 본부장이 제품 방향성을 이해할 수 있는 Mock 기반 시뮬레이터를 만드는 것이다.

핵심 컨셉은 다음과 같다.

좌측 패널: AI 회의 참석자

중앙 패널: 시스템 파이프라인 시각화

우측 패널: 샌드박스 실시간 렌더링

최종 단계: 승인 및 PRD 배포 Mock

좌측 패널은 단순 프롬프트 입력창이 아니라 AI PM이 회의 참석자처럼 사용자의 요청을 해석하고, 과거 GS/계룡 자료를 스캔하겠다고 제안하며, 사용자의 컨펌을 받는 구조여야 한다.

중앙 패널은 자산 DB화(RAG) → 3D OBJ 추출 Mock → 기획/디자인/개발/QA 에이전트 병렬 작업 → 샌드박스 렌더링 → 승인 대기 흐름을 로그 형태로 보여줘야 한다.

우측 패널은 ‘집뷰 옵션몰’ 예시를 기준으로 한다.
상단에는 타입 A/B 메뉴가 있고, 사용자가 바닥재/가전/조명/가구 버튼을 누르면 공간 Mockup의 상태가 실시간으로 변하는 구조여야 한다.

샌드박스 상단에는 [승인 및 PRD 배포] 버튼이 있으며, 필요한 분석과 테스트가 완료되면 활성화된다. 버튼 클릭 시 PRD 생성, 개발 계획 생성, 디자인 요청서 생성, QA 체크리스트 생성 Mock 흐름을 보여줘야 한다.

핵심은 사용자의 요청을 Orchestrator가 분석하고, 업무를 기획/개발/디자인/제작/QA 등 AI 본부장에게 분배한 뒤, 각 보고서를 취합해 샌드박스와 PRD/실행 계획으로 전환하는 구조입니다.

현재는 계룡 옵션몰 제작 케이스를 기준으로 구성했습니다.
GS 옵션몰 레퍼런스, 옵션 카탈로그, 정책서, 이미지 매핑 자료를 기반으로 업무 분해 → 에이전트 보고 → 샌드박스 확인 → 승인 및 PRD 배포 흐름을 확인할 수 있습니다.

개발 검토 요청드립니다.

검토 포인트:

1. 현재 플로우가 실제 구현 가능한 구조인지
2. Orchestrator/AI 본부장 역할 분리가 적절한지
3. 파일/URL 등록, 자산 검색, 보고서 생성, 샌드박스 빌드 구조를 어떻게 설계할지
4. 실제 DB/API/RAG 연동 시 필요한 개발 범위와 우선순위
5. MVP 기준에서 과한 기능과 먼저 구현해야 할 핵심 기능

1차 데모 기준으로 구현 가능성, 리스크, 필요한 개발 구조 검토 부탁드립니다.


추가로 자산 관리 정책도 함께 검토 부탁드립니다.

자산이 이미 있는 경우에는 기존 레포, 문서, 기획서, 회의록, 디자인 파일, 360 이미지, 3D 공간 데이터, 옵션 카탈로그 등을 검색해 Orchestrator가 각 AI 본부장에게 컨텍스트로 전달하는 구조가 필요합니다.

반대로 자산에 없는 경우에는 아래 흐름이 필요합니다.

1. 자산 없음 상태 표시
2. 필요한 자산 목록 자동 도출
3. 담당 AI 본부장별 제작/요청 항목 분리
4. 신규 촬영, 3D 제작, 디자인 제작, 정책 문서 작성, 데이터 입력 필요 여부 판단
5. 임시 Mock 또는 Placeholder로 샌드박스 구성
6. 필수 자산 미확보 시 승인/배포 차단

자산 DB화 작업도 별도 정의가 필요합니다.

검토 필요 항목:

* 파일/URL/레포/Figma/Drive 자료 등록 구조
* 자산별 메타데이터 정의: 프로젝트명, 유형, 출처, 등록일, 버전, 담당자, 사용 가능 여부
* 문서/회의록/PRD 텍스트 추출 및 검색 인덱싱
* 옵션 카탈로그/엑셀 데이터 구조화
* 이미지/360/3D 자산의 공간, 타입, 옵션, 레이어, 카메라 시점 매핑
* 기존 자산과 신규 요청 자산 구분
* RAG 검색용 벡터화 또는 키워드 검색 구조
* AI 본부장별로 어떤 자산을 참조했는지 기록
* 보고서/샌드박스/PRD에 사용된 자산 출처 표시
* 자산 누락, 버전 불일치, 파일명 불일치, 이미지 미노출 등 QA 체크

즉, 단순히 AI가 답변하는 구조가 아니라
“기존 자산을 찾아 재활용하고, 없는 자산은 제작 요청으로 분리하며, 모든 자료를 DB화해 이후 프로젝트에서도 다시 사용할 수 있는 구조”까지 포함해 개발 검토 부탁드립니다.
